## 一、PRD 核心信息提炼

**产品一句话**:类 Dify 的最小可用智能体编排平台,跑通"创建智能体 → 配置 → 对话 → 看日志"主链路,知识库选做。

**三套入口,十个大页面**:

| 入口 | 页面数 | 说明 |
|---|---|---|
| 官网前台 `www.xxx.com` | 1 | 产品介绍 + 登录 CTA |
| 用户控制台 `app.xxx.com` | 7 | 登录 / 智能体列表 / 配置 / 对话 / 会话详情 / 知识库 / 日志 |
| 管理后台 `admin.xxx.com` | 2 | 首页 / 用户与调用概览 |

**技术栈**:
- 前端:Next.js App Router + TypeScript + Tailwind + shadcn/ui
- 后端:Supabase(Auth / Postgres / Storage)+ 统一 LLM 适配层
- 部署:Vercel

**数据模型**:6 张核心表
`profiles / agents / chat_sessions / chat_messages / knowledge_documents / run_logs`

**两级角色**:
- 普通用户:管理自己的智能体、发起对话、看自己的日志
- 管理员:查看全平台用户和调用概览

## 二、5 个关键边界问题的反查结果

### Q1:多租户还是单用户?

**答案**:单用户,数据按 `user_id` 隔离

**依据**:
- §4 角色权限:"普通用户 = 管理自己的智能体"
- §9 业务规则:"用户只能访问自己的智能体和会话"
- §3 明确:"第一版不做多租户企业权限体系"

**实现方式**:每条数据带 `user_id`,用 Supabase RLS 做行级隔离。

---

### Q2:智能体复杂度?

**答案**:最简版本——Prompt + 模型 + 参数

**依据**:
- §5 配置页只有"名称/描述/Prompt/模型/参数/启停"
- §3:"第一版不做复杂工作流节点编排 UI、工具调用沙箱"

**不做**:workflow、function calling、多 Agent 协作

---

### Q3:知识库深度?

**答案**:最基础的文本上传 + 基础检索,且是可选功能

**依据**:
- §3:"可选知识库:仅支持文本文件上传与基础检索"
- §7 把知识库放在"第二优先级"
- §11 开发顺序里知识库排第 5 步

**⚠️ 矛盾点**:§12 把"第一版是否必须带知识库"列为待确认项,和 §3 有轻微矛盾。
**→ 最终决策**:MVP 不做,V2 实现(见《01-PRD-边界锁定.md》)。

---

### Q4:模型接入范围?

**答案**:PRD 原始文档未明确,§12 列为待确认项

**依据**:
- §1.0:"统一后端适配层对接第三方 LLM"(只说了有适配层,没说接谁)
- §6:"OpenAI 兼容接口"

**→ 最终决策**:MVP 接 DeepSeek,适配层按 OpenAI 兼容协议设计(见《01-PRD-边界锁定.md》)。

---

### Q5:是否要用户登录注册?

**答案**:要,用 Supabase Auth

**依据**:
- §1.0 技术选型直接指定 `Supabase Auth`
- §5 有独立登录页
- §11 开发顺序第 1 步就是"登录与基础鉴权"

## 三、PRD 的优点

1. **MVP 边界克制得很好**
 明确列出"第一版不做什么",这比"第一版要做什么"更有价值,能防止开发过程中范围蔓延。

2. **开发顺序已排好**(§11)
 对应敏捷迭代思路,可以直接作为开发路径参考。

3. **数据表结构完整**
 6 张表覆盖核心链路,字段设计合理。

4. **页面架构和用户路径图清晰**
 10 个大页面颗粒度刚好,mermaid 图示直观。

5. **技术选型统一**
 整套技术栈(Next.js + Supabase + Vercel)协同度高,适合学习和快速迭代。

## 五、数据模型的待调整项

基于边界决策,PRD §6 原始数据表需要做以下调整:

1. **`knowledge_documents` 表**:
 - PRD 原设计有 `agent_id` 字段 → 去掉,改用中间表
 - MVP 阶段不建此表,架构设计文档中定义结构
 - 拆分为 `knowledge_bases` + `documents` + `agent_knowledge_bindings` 三张表(V2)

2. **`run_logs` 表**:
 - 增加 `prompt_snapshot jsonb`
 - 增加 `response_snapshot text`

3. **`agents` 表**:
 - 不加 `knowledge_base_ids`(通过中间表关联)
 - 状态字段值明确:`draft` / `published` / `disabled`

详细数据模型见《04-数据模型设计.md》(架构设计阶段产出)。

## 六、风险提示

开发过程中需要持续关注的风险点:

1. **SSE 流式输出**:Next.js 流式 API 在 Vercel 部署时有边界情况(边缘函数超时等),需提前验证。

2. **Supabase RLS**:策略写错会导致数据泄漏或 API 401,需要有系统的测试方法。

3. **DeepSeek API 稳定性**:单一供应商,遇到限流或宕机会直接影响整个平台演示。适配层的意义就在于此,真出问题时能快速切换。

4. **日志数据膨胀**:保留完整 prompt 快照会让 `run_logs` 表快速膨胀,MVP 阶段数据量不大没问题,长期需要归档策略。

5. **多轮对话 token 超限**:超过模型上下文长度会报错,MVP 按条数截断是简单方案,长期需要更智能的策略。
